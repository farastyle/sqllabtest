# Laboratórios de Segurança (SQLi + XSS) - Instruções do Copilot

## Acesso ao Database

Para consultar o banco de dados Supabase, use o script `db-query.js`:

```powershell
node .\db-query.js "SELECT * FROM usuarios"
```

### Exemplos de Queries Úteis

```powershell
# Ver todas as tabelas
node .\db-query.js "\dt"

# Listar usuários
node .\db-query.js "SELECT * FROM usuarios"

# Listar produtos
node .\db-query.js "SELECT * FROM produtos"

# Inserir dados de teste
node .\db-query.js "INSERT INTO usuarios (email, senha, perfil) VALUES ('test@lab.com', 'senha123', 'user') RETURNING *"

# Listar comentários do mural (lab de XSS)
node .\db-query.js "SELECT * FROM mural_comentarios"
```

## Estrutura de Rotas

- `/` → Hub de seleção dos laboratórios (cards SQLi e XSS)
- `/sqli` → Login vulnerável a SQL Injection (tela original da primeira aula)
- `/dashboard`, `/produtos`, `/login`, `/test-payload`, `/reset` → rotas do lab de SQLi
- `/xss` → Dashboard do lab de XSS com os 10 testes na barra lateral
- `/xss/buscar` → Busca vulnerável a **XSS Refletido** (echo direto do `req.query.termo`)
- `/xss/mural` (GET/POST) → Mural vulnerável a **XSS Armazenado** (lê/grava `mural_comentarios`)
- `/xss/reset` → Restaura os comentários originais da tabela `mural_comentarios`

## Diretrizes de Desenvolvimento

- ✅ Manter vulnerabilidades intencionais:
  - SQLi via concatenação direta de string nas queries (`/sqli`, `/login`, `/produtos`, `/test-payload`)
  - XSS via renderização de HTML sem `escapeHtml()` nas rotas `/xss/buscar` e `/xss/mural`
- ✅ Centralizar código em `server.js`
- ✅ Usar `require('dotenv').config()` no início
- ✅ Incluir `ssl: { rejectUnauthorized: false }` no Pool do Supabase
- ✅ Usar variáveis de ambiente (`process.env.DB_ID`, `process.env.DB_SENHA`)
- ❌ Não usar queries parametrizadas nas rotas de SQLi (prepared statements quebrariam a vulnerabilidade)
- ✅ Pode usar queries parametrizadas no INSERT do mural (a vulnerabilidade ali é XSS, não SQLi)
- ❌ Não usar ORM (Sequelize, TypeORM)
- ❌ Não chamar `escapeHtml()` nas rotas `/xss/buscar` e `/xss/mural` — isso eliminaria a vulnerabilidade proposital

## Quando Ajudar

Se o usuário pedir:
- Criação de rotas vulneráveis → Manter SQLi ou XSS intencional, conforme o laboratório
- Testes de segurança → Usar `db-query.js` para verificar o estado do banco
- Documentação → Explicar os vetores de ataque
