# Laboratórios de Segurança (SQLi + XSS + IDOR) - Instruções do Copilot

## Acesso ao Database

Para consultar o banco de dados Supabase, use o script `db-query.js`:

```powershell
node .\db-query.js "SELECT * FROM usuarios"
```

### Exemplos de Queries Úteis

```powershell
# Ver todas as tabelas
node .\db-query.js "\dt"

# Listar usuários
node .\db-query.js "SELECT * FROM usuarios"

# Listar produtos
node .\db-query.js "SELECT * FROM produtos"

# Inserir dados de teste
node .\db-query.js "INSERT INTO usuarios (email, senha, perfil) VALUES ('test@lab.com', 'senha123', 'user') RETURNING *"

# Listar comentários do mural (lab de XSS)
node .\db-query.js "SELECT * FROM mural_comentarios"

# Listar perfis do lab de IDOR (sala 1)
node .\db-query.js "SELECT * FROM idor_perfis WHERE sala='1' ORDER BY numero"
```

## Estrutura de Rotas

- `/` → Hub de seleção dos laboratórios (cards SQLi, XSS e IDOR)
- `/sqli` → Login vulnerável a SQL Injection (tela original da primeira aula)
- `/dashboard`, `/produtos`, `/login`, `/test-payload`, `/reset` → rotas do lab de SQLi
- `/xss` → Seleção de sala (Lab 1 / Lab 2); `/xss/:sala` → dashboard com os 10 testes
- `/xss/:sala/buscar` → Busca vulnerável a **XSS Refletido** (echo direto do `req.query.termo`)
- `/xss/:sala/mural` (GET/POST) → Mural vulnerável a **XSS Armazenado**, isolado por sala
- `/xss/:sala/reset` → Restaura mural da sala e o catálogo de `produtos` (compartilhado com o SQLi)
- `/idor` → Seleção de sala (Lab 1 / Lab 2); `/idor/:sala` → dashboard com os 10 testes de IDOR
- `/idor/:sala/comprovante/:numero`, `/perfil/:numero`, `/perfil/:numero/editar`, `/admin/usuarios`,
  `/conta`, `/conta/editar`, `/api/perfil/:numero`, `/fatura/:codigo` (Base64), `/chamados`,
  `/chamado/:numero`, `/chamado/:numero/excluir`, `/pedidos` (via `?usuario_numero=`), `/mensagem/:numero`
  → as 10 telas vulneráveis a IDOR / quebra de controle de acesso, todas SEM checar se o
  registro pertence a quem está "logado" (perfil `numero=1` de cada sala)
- `/idor/:sala/validar` (POST) → compara a resposta do aluno com `RESPOSTAS_IDOR` (normalizada,
  ignorando acento/caixa/pontuação) e devolve `{ correto: true/false }`
- `/idor/:sala/reset` → restaura todas as tabelas `idor_*` da sala para o estado original
- `/misconfig` → login por sala (mesmas credenciais sala-a/a-sala e sala-b/sala-b do lab de IDOR,
  sessão separada `req.session.misconfigSala`); `/misconfig/:sala` → dashboard com os 10 testes
- `/misconfig/:sala/backup-loja.sql`, `/.env`, `/uploads`, `/uploads/relatorio-financeiro-2024.txt`,
  `/relatorio/:id`, `/painel-interno`, `/contato`, `/status`, `/diagnostico`,
  `/exportar/clientes.csv`, `/api/pedidos-internos` → as 10 telas/endpoints vulneráveis a
  Misconfiguration/exposição de dados sensíveis, todas SEM exigir login (a ausência de
  autenticação é a própria vulnerabilidade) e com conteúdo divergente por sala (`CONTEUDO_MISCONFIG`)
- `/misconfig/:sala/validar` (POST), `/misconfig/:sala/progresso`, `/misconfig/:sala/reset` →
  mesmo padrão de validação/progresso/reset do lab de IDOR, mas só a tabela `misconfig_progresso`
  é resetada (não há outro dado mutável neste lab)
- `/misconfig/painel-professor` → painel oculto, mesmo padrão do `/idor/painel-professor`

## Diretrizes de Desenvolvimento

- ✅ Manter vulnerabilidades intencionais:
  - SQLi via concatenação direta de string nas queries (`/sqli`, `/login`, `/produtos`, `/test-payload`)
  - XSS via renderização de HTML sem `escapeHtml()` nas rotas `/xss/:sala/buscar` e `/xss/:sala/mural`
  - IDOR via ausência de checagem de propriedade/papel em todas as rotas `/idor/:sala/*` que recebem
    um `:numero` (ou `?usuario_numero=`) — qualquer perfil pode ler/editar/excluir o registro de outro
- ✅ Centralizar código em `server.js`
- ✅ Usar `require('dotenv').config()` no início
- ✅ Incluir `ssl: { rejectUnauthorized: false }` no Pool do Supabase
- ✅ Usar variáveis de ambiente (`process.env.DB_ID`, `process.env.DB_SENHA`)
- ❌ Não usar queries parametrizadas nas rotas de SQLi (prepared statements quebrariam a vulnerabilidade)
- ✅ Pode usar queries parametrizadas no mural de XSS e em todas as tabelas `idor_*` (a vulnerabilidade
  ali não é SQLi, é XSS ou falta de controle de acesso)
- ❌ Não usar ORM (Sequelize, TypeORM)
- ❌ Não chamar `escapeHtml()` nas rotas `/xss/:sala/buscar` e `/xss/:sala/mural` — isso eliminaria a vulnerabilidade proposital
- ❌ Não adicionar verificação de "dono do registro" (`WHERE numero = usuarioLogado`) nas rotas de IDOR — isso eliminaria a vulnerabilidade proposital
- ✅ O progresso dos exercícios de IDOR e de Misconfiguration vem só do banco (tabelas `idor_progresso`
  e `misconfig_progresso`), nunca do `localStorage` — as respostas corretas (`RESPOSTAS_IDOR`,
  `RESPOSTAS_MISCONFIG`) nunca devem ser enviadas ao navegador antes da validação
- ❌ Não adicionar autenticação/checagem de permissão nas rotas `/misconfig/:sala/*` de conteúdo
  (backup, .env, uploads, painel-interno, status, diagnostico, exportar, api) — a ausência de
  autenticação É a vulnerabilidade proposital de cada exercício

## Quando Ajudar

Se o usuário pedir:
- Criação de rotas vulneráveis → Manter SQLi, XSS ou IDOR intencional, conforme o laboratório
- Testes de segurança → Usar `db-query.js` para verificar o estado do banco
- Documentação → Explicar os vetores de ataque
